# Task and context concepts

This document replaces the legacy Ariadne plan-DAG schema. Ari no longer treats a serialized plan DAG as the primary product model.

See also:

- `docs/ep/ari-workspace-runtime.md`
- `docs/adr/0002-workspace-as-runtime-unit.md`
- `docs/tool-projection.md`

## Current framing

Ari organizes LLM-assisted work around workspaces and runtime facts, not around a mandatory plan file.

Tasks, goals, prompts, proofs, context packets, and final responses may exist as structured records or projections, but they are subordinate to the workspace runtime model.

## Context packets

A context packet is an inspectable projection of selected workspace facts for an agent run or other operation. It may include:

- the user's goal or request;
- relevant folders or files;
- workspace command definitions;
- diff summaries;
- proof or command summaries;
- omitted facts and budget/selection notes where available.

Context packets should be generated by daemon-owned operations so clients and future agents can inspect or reproduce the same projection behavior.

## Tasks

A task is a useful runtime concept when work needs a goal, status, constraints, or acceptance anchors. A task is not required for every command or agent run, and this document does not define a durable task schema.

Future task schema work should be driven by real runtime needs and should stay workspace-scoped.

## Legacy plan DAGs

The old Ariadne plan schema described DAG-based execution artifacts. That model is historical and should not guide new Ari APIs unless a future decision explicitly revives part of it.
